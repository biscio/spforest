\section{Integration with \Spst}

The package \pkg{Rsandbox} relies on \spst to work.  
For the user, it means that a point pattern must be input as a \texttt{ppp} object 
and the covariates as \texttt{im} objects rather than rasters.
Specifically, \pkg{Rsandbox}  depends on  \pkg{spatstat.model},
\pkg{spatstat.geom}. In this document, 
we will additionally use the packages \pkg{spatstat.data} and \pkg{spatstat.random}.

\begin{Schunk}
\begin{Sinput}
R> library(Rsandbox)
R> library(spatstat.data)
R> library(spatstat.random)
\end{Sinput}
\end{Schunk}

\section{Getting started}

We need a point pattern as a \texttt{ppp} object, 
together with a list of covariates as list of \texttt{im} objects.
As an example we take the data \texttt{bei} and \texttt{bei.extra} from 
\spst. 
\begin{Schunk}
\begin{Sinput}
R> par(mfrow = c(1, 3), mar = c(0, 1, 0, 0))
R> plot(bei, pch = 16, cex = 0.5, main = "")
R> plot(bei.extra[[1]], ribbon = F, main = "")
R> plot(bei.extra[[2]], ribbon = F, main = "")
\end{Sinput}
\end{Schunk}
\includegraphics{Intro-004}

The function \texttt{RforestPP} is the core function of \pkg{Rsandbox} and 
generate random forest intensity estimates. 

\begin{Schunk}
\begin{Sinput}
R> forest <- RforestPP(
+    X = bei,
+    listcovariates = bei.extra,
+    Ntree = 3,
+    mtry = 1 / 3,
+    minpts = 500,
+    cores_trees = 3
+  )
\end{Sinput}
\end{Schunk}

The arguments: 
\begin{itemize}
  \item \texttt{Ntree} sets the number of intensity trees in the forest,
  \item \texttt{mtry} select the probability that a covariate is used to decide 
  a split,
  \item \texttt{minpts} is used as stopping criterion. Once the number of points 
  in a split is below \texttt{minpts}, we allow at most one last split. 
  This ensure that we allow regions with no points and 
  to not run the algorithm indefinitely,
  \item \texttt{cores{\textunderscore}trees} is used for parallelisation 
  where we allocate different intensity tree computation to different cores. 
\end{itemize}

Standard \R {} methods have been implemented: 
\begin{itemize}
  \item to plot the estimated random forest intensity,
\begin{Schunk}
\begin{Sinput}
R> plot(forest)
\end{Sinput}
\end{Schunk}
\includegraphics{Intro-006}
  \item to return the predicted intensity in new locations. There, the function 
  accepts the new locations to be a \texttt{ppp} object,
\begin{Schunk}
\begin{Sinput}
R> predict(object = forest, newdata = runifpoint(n = 5, win = bei$window))
\end{Sinput}
\begin{Soutput}
[1] 0.008481462 0.006127169 0.012198582 0.003744775 0.004985443
\end{Soutput}
\end{Schunk}
or a matrix with the $x$-coordinates on the first column and the $y$-coordinates 
on the second column.
\begin{Schunk}
\begin{Sinput}
R> predict(
+    object = forest,
+    newdata = cbind(
+      runif(5, min = 0, max = 1000),
+      runif(5, min = 0, max = 500)
+    )
+  )
\end{Sinput}
\begin{Soutput}
[1] 0.006127169 0.002238794 0.007694197 0.002238794 0.004985443
\end{Soutput}
\end{Schunk}
Behind the scene this is equivalent to return the estimated intensity 
as an \texttt{im} object and use \texttt{Extrat.im}. 
\begin{Schunk}
\begin{Sinput}
R> newdata <- runifpoint(n = 5, win = bei$window)
R> as.im(forest)[newdata]
\end{Sinput}
\begin{Soutput}
[1] 0.004985443 0.003286403 0.002238794 0.012503937 0.011560107
\end{Soutput}
\end{Schunk}
\end{itemize}


\section{A standard workflow}
\label{s:workflow}

When working with random intensity forest the workflow is a follow: 
\begin{enumerate}
  \item Choose a reasonable number of combinations of hyperparameters: 
  \texttt{minpts},  \texttt{mtry},
  \item Compute a random forest intensity for each combination 
  of the hyperparameters,
  \item Choose the combination of hyperparameters that maximises 
  the Out-Of-Bag (OOB) score,
  \item Re-run the random intensity forest with
  \texttt{Ntree} as large as possible,
  \item Use the variable importance (VIP) of the covariates 
  to assess their importance to the intensity of the process.
\end{enumerate}

Ideally, \texttt{Ntree} should be as large as possible 
but random intensity forest are computationally intensive. 
Thankfully, we have observed that 
the order of the VIP of each variables and the combination 
of hyperparameters that maximises the OOB score are usually the same 
as soon as $\texttt{Ntree} \geq 100$.   
Hence, if computational power is an issue, we suggest to choose 
$\texttt{Ntree} = 100$ in Step 2, and letting $\texttt{Ntree}$ 
to be as large as possible in Step 4. 

To illustrate this workflow on an example, 
we re-use the \texttt{bei} and \texttt{bei.extra} data. 
Further, to keep the computational burden low in this document, 
we choose a small number of combinations of hyperparameters: 
$\texttt{minpts} = 75, 100, 125$ 
and $\texttt{mtry}= 0.25, 0.5, 0.75$, with $\texttt{Ntree} = 6$. 
In practice, we should let $\texttt{minpts} = 1,2,\ldots,200$ 
and $\texttt{mtry}= 0.1,0.2,\ldots,1$, together with $\texttt{Ntree}\geq 100$ 
in Step 2 and $\texttt{Ntree}\geq 300$ in Step 4.

\begin{Schunk}
\begin{Sinput}
R> argmtry <- c(0.25, 0.5, 0.75)
R> argminpts <- c(seq(75, 125, by = 25))
R> allargs <- expand.grid(mtry = argmtry, minpts = argminpts)
\end{Sinput}
\end{Schunk}

We will use the \texttt{mapply} function. 



\begin{Schunk}
\begin{Sinput}
R> extrarg <- list(
+    Ntree = 4,
+    cores_tree = 2,
+    X = bei,
+    listcovariates = bei.extra
+  )
R> allforesttoy <- mapply(RforestPP,
+    mtry = allargs$mtry,
+    minpts = allargs$minpts,
+    MoreArgs = extrarg,
+    SIMPLIFY = F
+  )
\end{Sinput}
\end{Schunk}

We can now compute the OOB score of all the random intensity forests above 
and choose the one with the highest score. 
\begin{Schunk}
\begin{Sinput}
R> allscore <- lapply(allforesttoy, FUN = function(i) {
+    OOBscr(i, cores = 3)
+  })
R> allargs[which.max(allscore), ]
\end{Sinput}
\begin{Soutput}
  mtry minpts
2  0.5     75
\end{Soutput}
\end{Schunk}

Then, we run again \texttt{RforestPP} for this 
combination of parameter but with \texttt{Ntree} larger. 



\begin{Schunk}
\begin{Sinput}
R> chosenforesttoy <- RforestPP(
+    mtry = 0.5,
+    minpts = 75,
+    Ntree = 100,
+    cores_tree = 5,
+    X = bei,
+    listcovariates = bei.extra
+  )
\end{Sinput}
\end{Schunk}

Finally, we:
\begin{itemize}
\item Visualise the estimated intensity
\begin{Schunk}
\begin{Sinput}
R> plot(chosenforesttoy)
\end{Sinput}
\end{Schunk}
\includegraphics{Intro-016}
\item Assess the VIP of each variable
\begin{Schunk}
\begin{Sinput}
R> vipplot(chosenforesttoy, sorted = T)
\end{Sinput}
\end{Schunk}
\includegraphics{Intro-caching-viptoy}
\end{itemize}
Here there is no indication that the gradient or elevation are more important
than the other to explain the intensity of the point process. This was however 
a toy example, and the use of VIP shines when there is more covariates. 


\section{A more interesing example}

One of the advantage of random intensity forest is the ability 
to process a large number of covariates. 
We will re-study the point pattern \texttt{bei} but with a larger list of covariates, 
made available in \pkg{Rsandbox} in the list \texttt{beisoilres}.

\includegraphics{Intro-019}

From left to right and top to bottom, these covariates are: 
the elevation and gradient of the terrain, and the concentration of the soils 
in different nutrients, 
Al, B, Ca, Cu, Fe, K, Mg, Mn, P, Zn, N, Nmin, pH. 

We follow the workflow presented in Section~\ref{s:workflow}.
Once again, we only choose
a small number of combinations of hyperparameters to keep 
the computational burden light, 
but many more should be considered in practice. 

\begin{Schunk}
\begin{Sinput}
R> argmtry <- c(0.25, 0.5, 0.75)
R> argminpts <- c(seq(75, 125, by = 25))
R> allargs <- expand.grid(mtry = argmtry, minpts = argminpts)
R> extrarg <- list(
+    Ntree = 6,
+    cores_tree = 3,
+    X = bei,
+    listcovariates = beisoilres
+  )
R> allforestbei <- mapply(RforestPP,
+    mtry = allargs$mtry,
+    minpts = allargs$minpts,
+    MoreArgs = extrarg,
+    SIMPLIFY = F
+  )
\end{Sinput}
\end{Schunk}
Then, we find the combination of hyperparameters that maximises the OOB score.
\begin{Schunk}
\begin{Sinput}
R> allscore <- lapply(allforestbei, FUN = function(i) {
+    OOBscr(i, cores = 3)
+  })
R> allargs[which.max(allscore), ]
\end{Sinput}
\begin{Soutput}
  mtry minpts
3 0.75     75
\end{Soutput}
\end{Schunk}

Finally, we run again \texttt{RforestPP} with \texttt{Ntree} larger. 


\begin{Schunk}
\begin{Sinput}
R> chosenforestbei <- RforestPP(
+    mtry = 0.75,
+    minpts = 75,
+    Ntree = 100,
+    cores_tree = 2,
+    X = bei,
+    listcovariates = beisoilres
+  )
\end{Sinput}
\end{Schunk}

We can now visualise the estimated intensity and assess the VIP of each variable.
\begin{Schunk}
\begin{Sinput}
R> plot(chosenforestbei, main = "Estimated intensity")
\end{Sinput}
\end{Schunk}
\includegraphics{Intro-025}

There, the points in \texttt{bei} are very irregularly distributed
which results in the intensity being very high in two locations. 
Here a plot of the $\log$ of
the intensity is more revealing. 

\begin{Schunk}
\begin{Sinput}
R> plot(log(as.im(chosenforestbei)), main = "Logarithm of the estimated intensity")
\end{Sinput}
\end{Schunk}
\includegraphics{Intro-026}

Interestingly, we can assess the most covariates that had the 
most impact on the intensity of the point process. 

\begin{Schunk}
\begin{Sinput}
R> vipplot(chosenforestbei, sorted = T)
\end{Sinput}
\end{Schunk}

\includegraphics{Intro-caching-vipbei}

The VIP plot indicates that the five most important covariates 
are that grad, P, Zn, Cu, elev. 
Up to the covariates Cu, these results are ín agreement with 
those of \cite{coeurjolly:18} that studied the same data. 

Moreover, we have run the same analysis 
with all the combination of hyperparameters between $\texttt{mtry} = 0.1,0.2,\ldots 1$ and
$\texttt{minpts} = 10,20,\ldots,1000$ and 
$\texttt{Ntree} = 500$. 
The combination maximising the OOB score was 
$\texttt{mtry} = 1$ and $\texttt{minpts} = 10$ and the corresponding VIP plots 
showed the same five most important covariates. 
 



\section{More advanced uses}

The function \texttt{RforestPP} has more arguments that can be modified to train 
a random intensity forest. 
However, from our observations, they had a low or no impact on the mean integrated squared error and VIP. Consequently, we chose to not include them in the standard workflow.
These arguments are:
\begin{itemize}
  \item \texttt{score} which controls the score used to compare different splits. 
  \item \texttt{p} which controls the thinning applied on the original point pattern 
  \texttt{X} before training an intensity tree. If $p>0$, a $p$-thinning is applied, 
  and if $p=0$, the default, the points are bootstrapped. 
  \item  \texttt{minsplitq}, and \texttt{minsplitq}. By default we split a region
  according to the median level sets of the covariates, meaning that 
  $\texttt{minsplitq}=\texttt{minsplitq} = 0.5$ . A natural extension would 
  be to allow to random split a region according between the $100\texttt{minsplitq}\%$, 
  and $100\texttt{minsplitq}\%$ quantiles of the covariates' level sets.
  \item \texttt{threshold} which set the minimal area of a cells we allowed before attempting   one last split. By default, we set \texttt{threshold} to be 
  equal to the minimal pixel area of the images in \texttt{listcovariates} 
  as it makes no sense to go below the resolution of the covariates.
\end{itemize}
